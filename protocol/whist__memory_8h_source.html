109 /*
110 ==================
111 Region Allocator
112 ==================
113 
114 Note: All of these functions are safe, and will FATAL_ERROR on failure
115 
116 This is a thin wrapper around mmap/VirtualAlloc, and will give us fine-grained
117 control over regions of memory and whether or not the OS will allocate actual pages of RAM for us.
118 (Pages are usually size 4096, but are in-general OS-defined)
119 
120 This gives us fine-grained control over the "reported memory usage", which represents pages of
121 RAM that we demand must be allocated, and must not be written-to/read-from by any other processes.
122 This is what controls when the OS is "out of RAM", and what is reported by the OS's "Task Manager".
123 
124 The OS by-default uses lazy allocation. So we won't actually use RAM until we read or write
125 to a page allocated by allocate_region. All fresh pages we read from will be 0-ed out first
126 
127 Once we read/write to a page, that page becomes allocated in RAM. If you're done using it,
128 and you want to tell the OS that the data in the region isn't important to you anymore, call the
129 `mark_unused_region` function. That will decrease our reported memory usage by the size of the
130 region. However, the OS won't actually take those pages from us until it's low on RAM.
131 
132 If we want to tell the OS that the region is important to us again, call
133 `mark_used_region`. Just like allocate_region, this won't actually increase reported memory usage,
134 until we read/write to those pages. However, our pages will usually be left exactly as they were
135 prior to `mark_unused_region`. They'll only be zero-ed out if the OS needed those pages in the
136 meantime. This is much more efficient than having it be zero-ed out every time, it'll
137 only be zero-ed out if it was actually necessary to zero it out.
138 
139 If we want to completely give the region back to the OS, call `deallocate_region`,
140 and the RAM + address space will be freed.
141 
142 */
